The goal of this project is to explore, analyze, and visualize data while showcasing a variety of tools and techniques. A webpage version of the Jupyter notebook can be accessed here and a GitHub repository of the files can be found here.

Some questions that we will attempt to answer include...

  • Which complaints should we recommend to allocate more resources on resolving?
  • Is there a difference in average request closing time among the complaint types?
  • Is complaint type related to location?

First, let's import all the necessary initial libraries. Next, we will read the CSV file into a dataframe and print the shape to get an idea of how many rows and columns we are dealing with.

Let's use df.head() to get the first 5 rows of data and get our first glimpse into what the data actually looks like.

In this case we can already see that there are columns with NA values, but we may not always be able to see that just from df.head() since some datasets are very large, so we will comprehensively check for NA values using df.isna().sum(axis=0).

If we scroll through the list we can see that there are several columns that have a ton of NA values. Since columns with lots of NA values are not very useful, we can remove them by using df = df.loc[:, df.isna().sum() < 0.75*df.shape[0]], which removes all columns that have greater than 75% NA values. Now let's check for NA values again to confirm that certain columns are removed.

We can also take a look at the data types and identify any potential issues by using df.dtypes.

There are some data fields that should probably be date type, so we can use the astype() method to convert them to date type. Now let's check the data types again to confirm the change.

We can also check for weird date errors by making sure that there are no created dates that occur after the closed date by using df.loc[df['Created Date']>=df['Closed Date']].shape. The test passed so let's take a look at the city names to see if there are any potential issues here by using the unique() method for the city column.

There seems to be several cities that appear twice as both upper case and title case, so let's fix this by just making everything upper case through a function. Now we can check the city names again to confirm the fix.

Another way to do this is to use OpenRefine's facet feature to do this. First, we need to go the city column and choose facet then text facet.

Then we just need to edit the city names that are title case by converting them to all upper case. You can now extract this cleaned CSV version for use.

Let's start visualizing our data with a bar plot of the frequency of complaints for each city. We can see that Brooklyn has the most complaints, followed by New York and Bronx.

To explore Brooklyn a little more, let's create a dataframe of just Brookyln rows by using df_brooklyn=df[df['City'] == "BROOKLYN"]. In SQL, it would be something like SELECT * FROM myTable WHERE City='BROOKLYN'. Let's plot a scatter plot of complaint concentration in Brooklyn based on its latitude and longitude.

This scatter plot does not tell us very much as there is probably a lot of overlap, so let's try doing the same thing using a hexbin plot to better visualize density.

Instead of visualizing the number of complaints for cities, let's see how the number of different complaint types compare to each other by creating a bar plot for frequency of complaints. We can also use the head() method to get the top x number of complaints listed out.

Let's now explore the most common complaint in each city as well as its count.

It's pretty clear that blocked driveway and illegal parking are the most common complaints in most of these cities. Let's visualize this but just for Brooklyn again. Finally, let's explore the time it takes to close a complaint. Since we do not have such a column, we can create a time difference column and reformat it into a request closing time column.

Let's use the head() method to view our new columns.

Now that we have the request closing time column, we can visualize this using a bar plot to see which complaints take the longest to close on average.

The animal in a park complaint type seems like an outlier, so let's plot the bar plot again while ignoring that complaint.

The ferry complaint has no values so we can also exclude that. Let's create a new dataframe df2 for this.

Which complaints should we recommend to allocate more resources on resolving? Without the animal in a park complaint, derelict vehicle and graffiti complaints take the longest to close on average. So a recommendation could be to focus more on tackling derelict vehicle and graffiti problems. However, the frequency of graffiti complaints is very low as we saw from the complaint frequency chart, so focusing on resolving derelict vehicles is more important.

Is the complaint type related to location in df2? We can use a chi-square test of independence here by creating a contingency table with the crosstab() function and then feeding it into the test. Our null hypothesis is that there is no relation between complaint type and location and the alternative hypothesis is that there is a relation between complaint type and location.

Since our p-value is less than our alpha of 0.05, we reject the null hypothesis and conclude that there is a relation between complaint type and location.

Is there a difference in average request closing time among the complaint types in df2? Before we answer this, let's create a new dataframe that excludes animal in a park since it's an outlier. We can also exclude the ferry complaint as there are no time values. Let's also graph it to confirm the change. We will use this dataframe for our test.

We can use an ANOVA test for this where the null hypothesis is that there is no difference in average request closing time among the complaint types and the alternative hypothesis is that at least one complaint type differs significantly in average request closing time.

The p-value is less than our alpha of 0.05, so we reject the null hypothesis and conclude that at least one complaint type differs significantly in average request closing time.

An alternative way to do these tests is to use R, which is great for statistical testing. We can export df2 as CSV and use it here for the data sources. For the chi-square test, we can use the table() function to create the contingency table, and use chisq.test() for the test. We can see we get the same result as the Python method.

For the ANOVA test, we can use aov(). And we get the same results as before.
